# 产品说明

## Product

`live2love` 是一个“小手机”形态的 AI 聊天 PWA。项目名保持 `live2love`；手机主屏幕里的应用入口显示为 `聊天`。用户打开后，会进入一台拟真的私人手机：桌面、联系人、聊天、设置、记忆都像手机系统中的 App。核心能力是让用户配置自己的 API，与 AI 角色进行沉浸式长期聊天。

## Register

product

## Product Purpose

让非技术用户也能安全、清楚地把自己的模型 API 接入到一个沉浸式手机聊天环境里。产品要同时满足两件事：像酒馆一样可配置、可扮演、可长期维护角色；像一台小手机一样自然、私密、易打开。

## Users

主要用户：

- 想和 AI 角色长期聊天、陪伴、角色扮演的用户。
- 熟悉 SillyTavern、OpenRouter、本地模型或各类 API 服务，但希望有更轻、更手机化体验的用户。
- 不想把聊天数据托管在陌生服务里的用户。

次要用户：

- 想快速测试不同模型和角色设定的创作者。
- 想把 AI 联系人做成故事、世界观、互动剧的人。
- 想在手机上管理多个 AI 角色和聊天记录的人。

## Core Promise

你带来 API，它给你一台能聊天、能记住、能被你掌控的私人小手机。

## Strategic Principles

1. 私密感优先。聊天记录、角色设定、API Key 都应默认本地保存，敏感内容不做云端假设。
2. 配置要透明。Provider、Base URL、Model、Key、参数和连接错误必须用户可见、可测试、可恢复。
3. 沉浸不是遮蔽。模拟手机外壳增强体验，但不能隐藏错误、阻碍设置或降低可访问性。
4. 角色像联系人。人设、记忆、默认模型和对话关系应该从“联系人”心智出发，而不是从数据库字段出发。
5. 高级能力渐进暴露。世界书、系统提示词、上下文长度、采样参数等可以强大，但默认不压迫新用户。
6. 用户拥有出口。角色卡、聊天记录、记忆和设置都应支持导入、导出、清除。

## Product Shape

产品是一组手机 App，而不是单页聊天框。

- Lock Screen：时间、通知、最近消息、离线状态。
- Home Screen：App 图标、联系人小组件、最近会话。
- Messages：会话列表、聊天页、流式回复、消息操作。
- Contacts：AI 角色联系人、资料、关系、头像、默认模型。
- Characters：角色卡导入、提示词、人设、世界观。
- Memory：长期记忆、近期摘要、重要事件、用户偏好。
- Settings：API 配置、模型参数、主题、数据导入导出。

## MVP Scope

第一阶段必须完成：

- PWA 安装与离线壳。
- 模拟手机桌面和 Messages App。
- 用户配置 API Provider、Base URL、API Key、Model。
- 测试连接与错误提示。
- 至少一个 AI 联系人。
- 流式聊天、停止生成、重试。
- 本地保存聊天记录和 API 设置。
- 基础角色资料：名称、头像、简介、系统提示词。
- 设置页支持清除密钥和清除聊天数据。

第二阶段可以加入：

- 角色卡导入导出。
- 多联系人会话。
- 长期记忆编辑。
- 桌面通知。
- 锁屏通知入口。
- 多 Provider 预设。
- 上下文摘要。
- 主题和手机壳外观自定义。

## Non Goals

- 不做中心化聊天平台。
- 不默认代理用户 API 请求，除非后续明确加入后端。
- 不做公开角色市场作为 MVP。
- 不做复杂社交关系链。
- 不把参数配置做成面向开发者的控制台。
- 不用手机拟真遮盖产品未完成的基础能力。

## API Configuration

API 配置是产品信任的核心。

必须支持：

- Provider 名称。
- Base URL。
- API Key。
- Model。
- Temperature。
- Max Tokens 或等价参数。
- Stream 开关。
- 连接测试。

错误需要区分：

- Key 缺失。
- 认证失败。
- Base URL 无法连接。
- 模型不存在或无权限。
- 请求超时。
- Provider 返回限额或余额问题。
- 响应格式不兼容。

## Data And Privacy

默认策略：

- API Key 存在本地浏览器存储中，并在 UI 中默认隐藏。
- 聊天记录本地保存。
- 角色资料本地保存。
- 用户能一键导出和清除数据。
- 文档和界面都要明确说明：请求会发送到用户配置的 API 服务商。

如果未来加入后端或同步：

- 必须提供本地模式。
- 必须解释同步范围。
- 必须让用户选择是否上传聊天记录和角色数据。

## Tone

产品语气应像手机系统和私人助手之间的中间态。

推荐：

- 清楚
- 温柔
- 克制
- 可信
- 不装神秘

避免：

- 赛博玄学
- 过度二次元固定风格
- 技术黑话堆叠
- “AI 伴侣”式强绑定承诺
- 营销口吻

## UX Copy Principles

- 错误提示要告诉用户下一步怎么做。
- 设置项要给示例，不只给术语。
- 敏感动作要明确后果，例如清除聊天记录、删除角色、清除 API Key。
- 聊天界面的文案要短，避免破坏沉浸。
- 角色相关文案可以更有人味，API 相关文案必须准确。

## Anti References

不要像：

- 只有一个大输入框的普通 AI Chat。
- 密密麻麻的 API 调试器。
- 过度霓虹的 AI 工具。
- 套了手机边框的营销页面。
- 让所有设置都躲在不可发现菜单里的玩具 App。

可以学习：

- SillyTavern 的角色、预设、世界书和可配置性。
- iOS 的状态栏、桌面、通知和聊天心智。
- Telegram、iMessage、微信的会话可读性。
- Raycast 和 Linear 的清晰错误状态与设置效率。

## Success Criteria

用户第一次使用时：

- 3 分钟内能完成 API 配置并测试成功。
- 能清楚知道聊天请求会发送到哪里。
- 能创建或编辑第一个 AI 联系人。
- 能开始一次聊天，并在失败时知道如何修复。

长期使用时：

- 用户愿意把它安装在手机桌面。
- 用户能维护多个角色，而不迷失在参数里。
- 用户相信自己的数据可导出、可清除、可迁移。
- 聊天体验像私人手机，而不是网页表单。

## Product Risks

- 拟真手机过强，导致设置和错误处理难找。
- API 服务商差异大，容易出现响应格式兼容问题。
- 本地存储敏感信息需要清楚提示和安全边界。
- 长期聊天上下文会变大，需要摘要和记忆策略。
- 角色扮演需求多样，MVP 应避免一次性做完整酒馆替代品。
